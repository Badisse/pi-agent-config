1
Grill Yourself /skill:grill-with-docs
Interactive · 30-45 min

The agent interviews you relentlessly about your plan — one question at a time, walking down every branch of the decision tree. It explores the codebase to ground questions in reality.

What happens

  • Challenge your assumptions against existing code and domain language
  • Sharpen fuzzy terminology into precise canonical terms
  • Stress-test with concrete edge-case scenarios
  • Cross-reference your claims with actual code

Files produced

CONTEXT.md
Domain glossary — shared vocabulary for every future session
docs/adr/*.md
Architectural Decision Records — hard-to-reverse choices

Why it matters

The shared vocabulary makes every future session ~30% more token-efficient. This is the highest-leverage step — always start here.

2
Write the PRD /skill:to-prd
Interactive · 5 min

Synthesizes the grilling session into a Product Requirements Document. Published as a GitHub issue labeled needs-triage.

PRD structure

Problem Statement Solution User Stories (numbered, extensive) Implementation Decisions (modules, interfaces, schema) Testing Decisions (what to test, how) Out of Scope Further Notes

The PRD uses vocabulary from CONTEXT.md and respects ADRs — ensuring continuity from the grilling session.

3
Break into Vertical Slices /skill:to-issues
Interactive · 10-15 min

The PRD is broken into tracer bullet issues — each one a thin vertical slice through ALL layers (route → service → db → UI → test). Never horizontal (one layer only).

Each issue includes

  • Parent — link back to the PRD
  • What to build — end-to-end behavior description
  • Acceptance criteria — testable, behavioral checkboxes
  • Blocked by — dependency on other issues (or "can start immediately")
  • Type — HITL (needs human) or AFK (agent can do it)

Vertical vs Horizontal

✅ Vertical: "User can sign up" → route + form + validation + service + db + tests ❌ Horizontal: "Add all database models" → only schema, nothing demoable

Issues are published in dependency order (blockers first) so Blocked by references are real issue numbers.

4
Triage /skill:triage
Interactive · 10 min

Issues flow through a state machine. Each issue gets exactly one category (bug/enhancement) and one state label.

State machine

┌──────────────┐ │ unlabeled │ └──────┬───────┘ ▼ ┌──────────────┐ ┌────────────── │ needs-triage │ ──────────────┐ │ └──────┬───────┘ │ ▼ ▼ ▼ ┌────────────┐ ┌───────────────┐ ┌──────────────┐ │ wontfix │ │ needs-info │ │ ready-for- │ │ (closed) │ │ (waiting on │ │ human │ └────────────┘ │ reporter) │ └──────────────┘ └───────┬───────┘ │ (reporter replies) ▼ ┌──────────────┐ │ needs-triage │ (re-evaluate) └──────┬───────┘ ▼ ┌──────────────┐ │ ready-for- │ ← THIS IS THE GOAL │ agent │ └──────────────┘

For each ready-for-agent issue, the agent writes an agent brief — a behavioral contract specifying exactly what the autonomous agent should implement.

5
Ralph — Autonomous Implementation /ralph start N
Autonomous · 30-60 min

N parallel agents work in isolated Docker sandboxes, each in its own git worktree. Coordination through GitHub labels — no file conflicts, no collisions.

Architecture

/ralph start 3
Extension creates 3 worktrees + 3 tmux sessions
Agent A
1 Fetch ready-for-agent issues
2 Plan: read criteria, explore code, list behaviors
3 Claim issue → in-progress
4 Implement with TDD (red-green-refactor)
5 Feedback loops: test + typecheck + build
6 Commit + push + close issue
7 Reviewer: fresh agent reviews diff
8 Fix or reopen → loop to step 1
Agent B
1 Fetch ready-for-agent issues
2 Plan: read criteria, explore code, list behaviors
3 Claim issue → in-progress
4 Implement with TDD (red-green-refactor)
5 Feedback loops: test + typecheck + build
6 Commit + push + close issue
7 Reviewer: fresh agent reviews diff
8 Fix or reopen → loop to step 1
Agent C
1 Fetch ready-for-agent issues
2 Plan: read criteria, explore code, list behaviors
3 Claim issue → in-progress
4 Implement with TDD (red-green-refactor)
5 Feedback loops: test + typecheck + build
6 Commit + push + close issue
7 Reviewer: fresh agent reviews diff
8 Fix or reopen → loop to step 1
Coordination via GitHub labels — agents never share filesystem

Key safety mechanisms

  • Vertical slice validation — agent refuses horizontal issues, flags back to triage
  • Feedback loops required — tests + typecheck + build must pass before commit
  • Separate reviewer — fresh agent with no sunk-cost bias
  • Issue closed immediately — blockers resolve for other agents in real-time
  • Dependency-aware — agent skips blocked issues, picks them up in next iteration

Commands

/ralph start 3 Launch 3 parallel Docker agents /ralph status Show sessions + issues + worktrees /ralph log Tail latest session output /ralph stop Stop all sessions + clean worktrees /ralph recover Un-claim stale issues (only when no agents running)
6
Batch Merge /ralph merge
Autonomous + You Accept

After all agents finish, a merge agent creates a dedicated branch and reconciles everything. Your main branch is never touched automatically.

main
your base branch (untouched)
↓ forks from
agent/42-fix
agent/50-feat
agent/51-ref
↓ merge agent combines into
agent/merge-batch-2026-05-05
dedicated merge branch — tests run after each merge
↓ you review and accept
main
you merge when happy

What the merge agent does

  • Creates agent/merge-batch-<date> from your base branch
  • Merges each agent/* branch one at a time
  • Runs tests after each merge
  • Resolves conflicts if needed
  • Cleans up merged branches
  • Never touches the base branch or issue state
